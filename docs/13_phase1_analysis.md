# 1차 구현 결과 분석

## 분석 대상

현재 분석은 `data/sample_taxi_calls.csv`를 기반으로 한 오프라인 프로토타입 결과다.

즉, 실제 운영 데이터가 아니라 구조 검증용 샘플 데이터 기준 결과이며, 현재 단계의 목적은 다음과 같다.

- 전처리 파이프라인이 정상적으로 동작하는지 확인
- 기본적인 수요 변화가 잡히는지 확인
- 수요/공급 기반 배차 우선순위 계산이 가능한지 확인

## 전처리 결과

생성된 전처리 결과 파일에는 다음 컬럼이 포함된다.

- `pickup_datetime`
- `zone_id`
- `call_count`
- `available_taxis`
- `pickup_latitude`
- `pickup_longitude`
- `hour`
- `day_of_week`
- `lag_1`
- `rolling_mean_3`

즉, 원본 호출 데이터에 시간 피처와 시계열 피처가 추가되었다.

## 현재 샘플 데이터 요약

전체 행 수:
- 12 rows

지역별 총 수요:
- A 지역: 127
- B 지역: 61

시간대별 총 수요:
- 18시: 70
- 19시: 118

## 해석

### 1. 시간대별 수요 증가

19시의 총 호출 수는 118로, 18시의 70보다 크다.

이 결과는 현재 샘플 데이터 기준으로 저녁 시간이 되면서 전체 수요가 증가하는 구조를 보여준다.

### 2. 지역별 수요 편차

A 지역의 총 수요는 127, B 지역은 61로 약 2배 이상 차이가 난다.

즉, 현재 샘플에서는 A 지역이 핵심 수요 집중 지역으로 해석할 수 있다.

### 3. 최신 시점 수급 불균형

최신 시점 기준 imbalance score:

- A 지역: `30 / 2 = 15.0`
- B 지역: `13 / 4 = 3.25`

이 결과는 A 지역의 수요 대비 공급 부족이 훨씬 심하다는 뜻이다.

따라서 현재 rule-based dispatch 기준에서는 A 지역을 우선 배차 대상으로 판단하는 것이 타당하다.

## 현재 그래프 의미

`outputs/hourly_demand.png`는 시간대별 총 호출 수를 시각화한 막대그래프다.

이 그래프는 현재 샘플 데이터에서:
- 18시보다 19시에 수요가 더 높고
- 따라서 시간대에 따라 사전 배차 강도를 다르게 둘 필요가 있음을 보여준다.

## 1차 구현에서 의미 있는 점

- 데이터 전처리 구조가 분리되어 있다.
- 피처 생성, 예측, 배차, 시각화가 각각 독립 모듈로 분리되어 있다.
- 단순한 수식으로도 배차 우선순위를 설명할 수 있다.
- 발표 시 “왜 A 지역에 더 배차해야 하는가?”를 숫자로 설명할 수 있다.

## 현재 한계

### 1. 데이터가 너무 작다

현재 샘플 데이터는 단지 구조 검증용이라서 실제 예측 성능을 논하기에는 부족하다.

### 2. baseline 모델 해석 한계

현재 모델은 RandomForest baseline이며, 데이터 규모가 작아 성능 수치 자체보다 파이프라인 동작 여부 확인의 의미가 더 크다.

### 3. 시간 분할이 아직 단순하다

현재는 일반적인 train/test split 방식이므로, 이후 실제 프로젝트에서는 시계열 순서를 반영한 split으로 바꾸는 것이 맞다.

### 4. 외부 변수 미반영

현재는 날씨, 이벤트, 공휴일, 교통상황 등 외부 변수를 반영하지 않았다.

## 다음 개선 우선순위

1. 공개 택시 데이터셋 확보
2. 시간 순서를 반영한 학습/검증 분리
3. 지역 단위 고정(zone 또는 grid)
4. 외부 변수 추가
5. 배차 로직 시나리오 테스트
6. Unity 시각화와 연결

## 결론

현재 1차 구현은 “실제 서비스 수준의 정확한 예측 시스템”이 아니라,

`수요 예측 -> 수급 불균형 계산 -> 우선 배차 판단`

이라는 전체 구조가 성립하는지 확인하는 오프라인 프로토타입이다.

이 단계는 성공적으로 완료되었다고 볼 수 있으며, 다음 단계는 실제 공개 데이터셋 기반으로 이 구조를 확장하는 것이다.
